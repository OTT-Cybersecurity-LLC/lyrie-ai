## Theme

**v1.1 is the release that makes Lyrie a complete drop-in replacement for OpenClaw.**

Run your entire operation — Telegram bot, crons, skills library, memory, multi-agent orchestration — on Lyrie, with all of Lyrie's security capabilities on top.

## Migration Path (for operators coming from OpenClaw)

```
Step 1: Install Lyrie
  curl -fsSL https://lyrie.ai/install.sh | bash
  lyrie doctor  # verify dependencies

Step 2: Migrate configuration
  lyrie migrate --from openclaw
  # Migrates: channels, MEMORY.md, USER.md, SOUL.md, AGENTS.md, skills (metadata)

Step 3: Migrate memory
  lyrie migrate memory --from openclaw --source ~/.openclaw/memory/lancedb
  # Converts: LanceDB vectors → Lyrie SQLite FTS5 + optional LanceDB layer

Step 4: Migrate skills
  lyrie migrate skills --from openclaw --source ~/.openclaw/workspace/skills/
  # Indexes: 129 SKILL.md files → Lyrie skill registry (readable + runnable)

Step 5: Migrate cron jobs
  lyrie migrate crons --from openclaw
  # Reads: openclaw runtime cron API → writes to ~/.lyrie/crons.json

Step 6: Configure workspace context
  lyrie workspace init --import ~/.openclaw/workspace/
  # Sets: SOUL.md, AGENTS.md, MEMORY.md as always-injected context files

Step 7: Install as background service
  lyrie service install   # launchd on Mac, systemd on Linux
  lyrie service start
  lyrie status

Step 8: Verify
  lyrie doctor --full
  lyrie cron list
  lyrie skills list | wc -l   # should show 129+
  lyrie memory search "Guy Sheetrit"

Step 9: Test bot integration
  # Send message to Lyrie bot on Telegram → verify response
  lyrie channel test telegram

Step 10: Cut over
  # Stop OpenClaw daemon
  openclaw gateway stop   # (Guy does this manually)
  # Lyrie takes over all channels from here
```

## v1.1 Feature List (prioritized)

### P0 — Required for OpenClaw parity (blockers for cutover)

#### P0-1. `lyrie cron` CLI — OpenClaw cron tool parity
**What:** Full CLI that matches OpenClaw's `cron` tool interface.  
**Why needed:** OpenClaw has ~63 crons (~39 enabled) driving everything from heartbeat to SEO snapshots to memory backups. Without `lyrie cron list/add/update/remove/run`, operators can't migrate their automation.  
**Technical approach:**
- Add `lyrie cron` subcommand to the CLI router
- Commands: `list`, `add <name> --schedule <cron> --task <...>`, `update <id>`, `remove <id>`, `run <id>`, `status`, `enable <id>`, `disable <id>`
- Backend: extend `CronManager` with CRUD persistence to `~/.lyrie/crons.json`
- Import: `lyrie migrate crons --from openclaw` reads OpenClaw's runtime API (port 18789) or falls back to `~/.openclaw/crons.json` if server offline
- **Complexity: 4 days**
- **Success criteria:** `lyrie cron list` shows all imported crons; `lyrie cron run <id>` executes manually; crons survive daemon restart

#### P0-2. SKILL.md runtime — execute OpenClaw-format skills
**What:** Lyrie's SkillManager must be able to load and execute SKILL.md-format skills, not just JSON skills.  
**Why needed:** OpenClaw's 129 skills are all SKILL.md format. The current migrator copies skill metadata but Lyrie cannot RUN them. The entire skills ecosystem is unusable post-migration.  
**Technical approach:**
- `packages/core/src/skills/skill-md-loader.ts` — parses SKILL.md format (frontmatter + markdown instructions)
- `packages/core/src/skills/skill-md-executor.ts` — executes SKILL.md as agent context injection (same mechanism as OpenClaw)
- `lyrie skills list --format md` — lists all SKILL.md skills indexed from registered directories
- `lyrie skills run <skill-name>` — executes a skill by injecting its SKILL.md into the next agent turn
- Registry: `~/.lyrie/skill-dirs.json` tracks directories to scan (defaults to `~/.lyrie/skills/` + imported paths)
- **Complexity: 5 days**
- **Success criteria:** `lyrie skills run ga4-analytics` works; `lyrie migrate skills --from openclaw` indexes all 129; skill is injected into context exactly like OpenClaw

#### P0-3. Memory migration — LanceDB → Lyrie
**What:** `lyrie migrate memory --from openclaw` that reads OpenClaw's LanceDB (`~/.openclaw/memory/lancedb/memories.lance`) and imports into Lyrie's SQLite FTS5.  
**Why needed:** OpenClaw's vector memories are the accumulated context from months of operation. Current migrator only reads MEMORY.md as plain text lines — loses all structured vector memories.  
**Technical approach:**
- LanceDB Node.js reader: use `@lancedb/lancedb` npm package to scan the `memories.lance` table
- Map: LanceDB columns (`id`, `text`, `category`, `importance`, `tags`, `createdAt`, `vector`) → Lyrie's `LyrieMemoryEntry`
- Vectors: store raw float32 arrays in a new `memory_vectors` table (for future LanceDB layer in Lyrie)
- `lyrie migrate memory --from openclaw --dry-run` shows what would be imported
- **Complexity: 3 days**
- **Success criteria:** `lyrie memory search "SEO"` returns results from the OpenClaw vector store post-migration

#### P0-4. Workspace context injection — SOUL.md / AGENTS.md / MEMORY.md always-on
**What:** Lyrie must inject operator workspace files into EVERY agent conversation, exactly like OpenClaw's "Project Context" system.  
**Why needed:** OpenClaw injects SOUL.md + AGENTS.md + MEMORY.md + USER.md into every turn's system prompt. Currently, Lyrie only stores these as memory entries (not injected as context). This means the agent doesn't have its operating rules, personality, or memory index in every conversation.  
**Technical approach:**
- `packages/core/src/engine/workspace-context.ts` — reads a configurable workspace dir (default `~/.lyrie/workspace/`)
- Ordered injection: SOUL.md → IDENTITY.md → USER.md → AGENTS.md → MEMORY.md → any `*.md` in workspace root
- Size limit: inject up to 12,000 chars per file (warn if truncated, like OpenClaw does)
- `lyrie workspace init --import <path>` copies files from OpenClaw workspace
- `lyrie workspace list` shows what will be injected each turn
- **Complexity: 3 days**
- **Success criteria:** After `lyrie workspace init --import ~/.openclaw/workspace/`, every Lyrie response reflects SOUL.md persona and AGENTS.md rules

#### P0-5. `sessions_spawn` equivalent — subagent orchestration tool
**What:** Expose Lyrie's SubAgentManager as a first-class tool that the LLM can call, equivalent to OpenClaw's `sessions_spawn`.  
**Why needed:** OpenClaw's spawn-first architecture (any task >3 tool calls → spawn subagent) is a core operational pattern. Guy relies on it for all heavy work. Lyrie has SubAgentManager internally but the LLM can't call it directly.  
**Technical approach:**
- New tool: `spawn_subagent` in the Lyrie tools catalog
  ```json
  {
    "name": "spawn_subagent",
    "description": "Spawn an isolated subagent for a specific task. Returns task_id. Results delivered via push completion.",
    "params": { "task": "string", "model": "string?", "context": "string?", "lightContext": "boolean?" }
  }
  ```
- Wire to existing `SubAgentManager.spawn()` in `packages/core/src/agents/sub-agent.ts`
- Expose `sessions_list` equivalent: `list_subagents` tool
- **Complexity: 3 days**
- **Success criteria:** LLM can call `spawn_subagent` and results are delivered back into the conversation; coordinator mode (already in codebase) gates on this

#### P0-6. launchd/systemd service install
**What:** `lyrie service install` creates a launchd plist (Mac) or systemd unit (Linux) and starts Lyrie as a persistent background service.  
**Why needed:** OpenClaw runs as a launchd-managed daemon (`~/.openclaw/...`). Lyrie must be equally persistent — survive reboots, auto-restart on crash, keep all 13 channel connections alive.  
**Technical approach:**
- `scripts/service.ts` — CLI subcommand `lyrie service install|uninstall|start|stop|status|logs`
- Mac: generates `~/Library/LaunchAgents/ai.lyrie.agent.plist` pointing to `lyrie daemon`
- Linux: generates `~/.config/systemd/user/lyrie.service`
- Both: set `LYRIE_DAEMON_PORT`, log to `~/.lyrie/logs/daemon.log`, auto-restart on crash (keepalive=true / Restart=always)
- **Complexity: 2 days**
- **Success criteria:** `lyrie service install && lyrie service start` → `lyrie status` shows daemon running; survives `reboot` test (or launchd unload/reload simulation)

#### P0-7. Migration completeness — cron jobs + workspace files
**What:** Fix `lyrie migrate --from openclaw` to correctly migrate cron jobs (currently looks for `crons.json` which doesn't exist — OpenClaw crons are in runtime DB) and workspace context files.  
**Why needed:** Current migrator has a bug — it looks for `~/.openclaw/crons.json` but OpenClaw stores crons in its runtime database (accessed via port 18789 API). Zero crons are migrated today.  
**Technical approach:**
- Add fallback: if `crons.json` missing, try `GET http://localhost:18789/api/v1/crons` (openclaw must be running), else warn and skip with instructions
- Add workspace migration step: copy SOUL.md, AGENTS.md, MEMORY.md, USER.md, IDENTITY.md, HEARTBEAT.md to `~/.lyrie/workspace/`
- **Complexity: 2 days**
- **Success criteria:** `lyrie migrate --from openclaw` with OpenClaw running migrates all 63 crons + all workspace files with zero manual steps

### P1 — Security + ATP advancement

#### P1-1. XBOW benchmark — run and publish
**What:** `lyrie benchmark xbow` runs the 104-challenge XBOW validation suite and publishes results to `lyrie.ai/research/benchmarks`.  
**Why needed:** Credibility. PentestGPT claims 86.5% (USENIX). XBOW claims 100% on their own suite. Lyrie needs a published score to compete.  
**Technical approach:**
- `scripts/benchmark-xbow.ts` — clone XBOW challenge repo, run Lyrie against each challenge, score
- GitHub Action: run benchmark on each release, publish result JSON to `research.lyrie.ai/benchmarks`
- **Complexity: 4 days**

#### P1-2. ATP IETF draft — format and submit
**What:** Format `packages/atp/` as an IETF Internet-Draft and submit.  
**Why needed:** The ATP (Agent Trust Protocol) is a significant IP asset. An IETF draft establishes precedence and credibility for Lyrie's trust model.  
**Technical approach:**
- Convert `packages/atp/README.md` → IETF RFC format using `xml2rfc`
- Submit as `draft-ottcyber-lyrie-atp-00`
- **Complexity: 3 days**

#### P1-3. OSS-Scan — deploy to EPYC
**What:** Deploy the Dockerized OSS-Scan service to EPYC, expose at `lyrie.ai/research/scan`.  
**Why needed:** `deploy/oss-scan/` has a complete Dockerfile + docker-compose but nothing is running on EPYC (verified). This is a promised public feature.  
**Technical approach:**
- `scp -r deploy/oss-scan/ root@104.192.2.18:/root/lyrie-oss-scan/`
- `ssh epyc "cd /root/lyrie-oss-scan && docker-compose up -d"`
- Add nginx reverse proxy route for `lyrie.ai/research/scan` → `localhost:3020`
- Uptime monitor via existing heartbeat cron
- **Complexity: 1 day**

#### P1-4. LanceDB as optional Lyrie memory layer
**What:** Add optional LanceDB vector layer to Lyrie's memory for semantic search (on top of current FTS5).  
**Why needed:** OpenClaw uses LanceDB for semantic search. Without it, migrated vector memories are stored but not semantically queryable.  
**Technical approach:** Add `@lancedb/lancedb` as optional peer dep; `packages/core/src/memory/layers/lancedb-layer.ts`  
**Complexity: 3 days**

#### P1-5. Multi-channel inbox expansion (beyond telegram + whatsapp)
**What:** Enable the remaining 11 channel adapters Lyrie has built (Discord, Slack, Signal, Matrix, IRC, etc.) via `lyrie channel enable <name>`.  
**Why needed:** OpenClaw supports 23+ channels. Lyrie has the adapters (13 in gateway.ts) but only telegram and whatsapp are configured for Guy's installation. v1.1 should make all channels easy to activate.  
**Technical approach:** `lyrie channel list --available` / `lyrie channel enable discord --token <...>` wizard  
**Complexity: 2 days**
